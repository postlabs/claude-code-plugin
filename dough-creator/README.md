# dough-creator

Claude Code plugin that builds **Toast automations** from natural-language
requests. It is the kit-builder role, automated: where the in-app OVEN agent
can only compose existing capabilities, this plugin can also create the
capabilities themselves — full kits with Python tools.

```
/create  "when a new video drops on this channel, summarize it to Telegram"
/test    # register into Toast + bake-verify (repairs failures in place)
/publish # deploy the verified automations
```

## Three commands — a build → test → deploy pipeline

| Command | Step | Needs Toast? | Does |
|---------|------|--------------|------|
| **`/create`** | build | no | Author the artifacts in `./<slug>/`; static-validate + unit-run them. Never bakes. Identical online or off. |
| **`/test`** | test | **yes** | Register the workspace into Toast, bake the root doughs, and REPAIR failures in place (fix → re-register → re-bake) until green. Stamps provenance `verified`. |
| **`/publish`** | deploy | **yes** | Deploy ONLY `verified` artifacts (test gates deploy) and hand over the management controls. |

The split keeps each command's job legible — and because `/test` owns the
bake-and-repair loop end to end, separating it from `/create` costs no
quality: the feedback loop moved into `/test`, it did not break.

## How it works

- **Toast is the runtime, the plugin is the builder.** Registration, binding,
  and baking are done by the locally running Toast backend (port 18587) during
  `/test` and `/publish`. The plugin authors files and drives the backend's
  APIs.
- **cwd is the source of truth.** Each `/create` run authors everything
  under `./<automation_slug>/` in the session's working directory — the user
  can see, version, and edit what was created. Toast holds only the
  published runtime copies; the plugin never writes into Toast profile
  directories and never guesses profile paths.

  ```
  ./<slug>/
  ├── kits/<kit_id>/          ──kit_lifecycle.py install──▶  backend copies into its profile
  │                              edit loop: edit here → reload
  └── doughs/<dough_slug>/    ──dough_publish.py publish──▶  POST/PUT /doughs
                                 modify loop: pull → edit → publish
  ```

- **peel MCP** (vendored under `vendor/peel/`, talks HTTP to the backend) for
  discovery / validation / baking. Two API surfaces peel does not cover get
  script wrappers: the **kit lifecycle API** (install / hot-reload /
  uninstall, `scripts/kit_lifecycle.py`) for binding new Python kits without
  a backend restart, and the **dough CRUD API**
  (`scripts/dough_publish.py` — publish / pull / delete) for registering
  user doughs. Publishing IS validation: a 422 response body carries the
  `validation_errors` to fix before republishing.
- **Routing:** existing flours → composition · reasoning gap → user agent
  flour · reach gap → new kit · browser gap → action-creator plugin.

## `/create` works without Toast

The build step needs no backend. `offline_validate.py` runs the REAL engine
validator — vendored under `vendor/engine_core/` (a version-stamped slice,
pydantic-only) — so parse + composition rules + box checks happen locally,
and `tool_runner.py` unit-runs kit tools against a stubbed `_core`. Two
verification levels, recorded per artifact in `./<slug>/provenance.yaml`:

| Level | Reached by | Means |
|-------|-----------|-------|
| **`static`** | `/create` (no Toast) | Authored + static-validated + unit-run. NOT yet run on the real engine. Not deployable. |
| **`verified`** | `/test` (Toast up) | A real root-dough bake ran green. Deployable by `/publish`. |

- **Discovery** still needs the backend (peel): offline, `/create` composes
  against workspace artifacts + floor capabilities only and flags external
  assumptions as warnings.
- **Drift stamp.** The vendored engine slice carries a version stamp (mojo git
  rev + sync date), regenerated by `sync_engine_core.py` (which READS the mojo
  repo and re-runs a smoke test) — never hand-edited.

## Requirements

You bring the Python. Everything runs on the interpreter on your `PATH` (or one
you point `TOAST_PYTHON` at) — the plugin never assumes a Toast/mojo checkout.
What each command needs of it:

- **`/create` (build):** `pydantic` + `ruamel.yaml` — offline validation and
  tool unit runs. No Toast.
- **`/test` + `/publish`:** the Toast app installed and **running**, plus
  `mcp` + `httpx` + `httpx-sse` on the interpreter (the peel MCP server). If a
  dep is missing the relevant tools just don't work, with a message saying
  what to install (`pip install pydantic ruamel.yaml mcp httpx httpx-sse`).

`run_peel.cmd` resolves the interpreter as `TOAST_PYTHON` env → `python` on
PATH. Set `TOAST_PYTHON` to a Toast install's bundled Python if you'd rather
not install the deps yourself.

## Layout

```
commands/create.md            /create  — BUILD: author + static/unit checks (no backend)
commands/test.md              /test    — TEST: register + bake-verify + repair loop (Toast up)
commands/publish.md           /publish — DEPLOY: ship verified artifacts + hand over controls
skills/kit-authoring/         kit contract (tools.py rules, kit.yaml, lifecycle loop)
skills/dough-authoring/       dough/flour authoring (fetches Toast's live build guide)
scripts/toast_env.py          preflight: backend up? (connected/standalone) + profile diagnostics
scripts/offline_validate.py   BUILD: parse + static rules + box, vendored engine, no backend
scripts/tool_runner.py        BUILD: kit-tool unit runs (stubbed _core), no backend
scripts/kit_lifecycle.py      TEST/DEPLOY: /kits install · reload · uninstall · list wrapper
scripts/dough_publish.py      TEST/DEPLOY: user-dough publish · pull · delete (API-only)
scripts/sync_engine_core.py   dev: regenerate the vendored engine-validator slice from the mojo repo
vendor/peel/                  vendored peel MCP server (stdio)
vendor/engine_core/           vendored engine-validator slice (version-stamped, pydantic-only)
vendor/core_stub/             stubbed _core for offline kit-tool unit runs
```

## v1 scope

- Auth-less kits only (`auth: type: none`) — pure compute, public APIs, local
  files. OAuth kits (connect flows, console app registration) are out.
- Kits land as third-party kits in the local profile (copied there by the
  install API); the cwd source stays the permanent home. Promotion to
  official bundled kits (Toast repo, hashes, review) is a separate manual
  step.
- Web/browser automations are delegated to the action-creator plugin.
